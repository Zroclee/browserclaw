在 Express + Node.js + TypeScript 项目中，实现统一的返回格式和全局异常处理可以极大提升 API 的规范性、可维护性和用户体验。下面我将详细介绍如何从零搭建这一套机制。

---

## 1. 定义统一的响应结构

首先，我们定义一个标准的响应体格式。通常包括状态码、消息和数据。

```typescript
// types/response.ts
export interface ApiResponse<T = any> {
  code: number;       // 业务状态码，0 表示成功，非 0 表示错误
  message: string;    // 提示信息
  data?: T;           // 成功时返回的数据
  error?: any;        // 错误时携带的详细信息（可选）
}
```

为了在代码中方便使用，我们可以扩展 Express 的 `Response` 对象，添加 `success` 和 `error` 方法。

```typescript
// types/express.d.ts
import { ApiResponse } from './response';

declare global {
  namespace Express {
    interface Response {
      success<T>(data?: T, message?: string): Response;
      error(code: number, message: string, error?: any): Response;
    }
  }
}
```

然后实现这些方法，可以在一个中间件中挂载到 `res` 上。

---

## 2. 创建统一响应中间件

编写一个中间件，将 `res.success` 和 `res.error` 方法注入到每个请求的响应对象中。

```typescript
// middlewares/responseFormatter.ts
import { Request, Response, NextFunction } from 'express';

export function responseFormatter(req: Request, res: Response, next: NextFunction) {
  // 注入成功响应方法
  res.success = function<T>(data?: T, message: string = 'Success') {
    return this.json({
      code: 0,
      message,
      data,
    });
  };

  // 注入错误响应方法
  res.error = function(code: number, message: string, error?: any) {
    return this.status(200).json({  // 注意：HTTP 状态码通常保持 200，业务错误码由 code 区分
      code,
      message,
      error,
    });
  };

  next();
}
```

> 说明：`res.error` 中设置 `status(200)` 是为了让 HTTP 状态码始终为 200，业务错误通过 `code` 区分。如果希望根据错误类型设置不同的 HTTP 状态码，也可以调整。

## 3. 全局异常处理中间件

在 Express 中，错误处理中间件需要放在所有路由之后，并且有四个参数（err, req, res, next）。它会捕获路由处理中通过 `next(err)` 传递的错误，以及同步代码中抛出的错误（需要配合 `express-async-errors` 处理异步错误）。

### 3.1 安装 `express-async-errors`（可选）

该库可以自动捕获异步路由中抛出的异常，无需手动 `try-catch`。

```bash
npm install express-async-errors
```

然后在入口文件顶部引入：

```typescript
import 'express-async-errors';
```

### 3.2 编写错误处理中间件

```typescript
// middlewares/errorHandler.ts
import { Request, Response, NextFunction } from 'express';

// 自定义业务异常类（可选）
export class BusinessError extends Error {
  public code: number;
  constructor(code: number, message: string) {
    super(message);
    this.code = code;
    this.name = 'BusinessError';
  }
}

export function errorHandler(err: any, req: Request, res: Response, next: NextFunction) {
  // 记录错误日志（可根据需要替换为更完善的日志系统）
  console.error(`[Error] ${err.message}`, err);

  // 判断是否为业务异常
  if (err instanceof BusinessError) {
    return res.error(err.code, err.message);
  }

  // 其他未知错误，返回 500
  return res.error(500, 'Internal Server Error', process.env.NODE_ENV === 'development' ? err.message : undefined);
}
```

> 注意：`res.error` 方法已在 `responseFormatter` 中定义，所以可以直接使用。

## 4. 处理未捕获的异常和 Promise rejection

在 Node.js 中，未被捕获的异常或未处理的 Promise rejection 会导致进程退出。我们需要添加全局监听，避免应用崩溃，同时优雅地记录错误。

```typescript
// 在入口文件（如 app.ts 或 server.ts）中添加
process.on('uncaughtException', (err) => {
  console.error('Uncaught Exception:', err);
  // 可根据需要执行一些清理工作，然后退出进程
  process.exit(1);
});

process.on('unhandledRejection', (reason, promise) => {
  console.error('Unhandled Rejection at:', promise, 'reason:', reason);
  // 这里不建议直接退出，可记录后继续运行，但应当确保应用状态正常
});
```

## 5. 在应用中集成中间件

在 Express 应用实例中，按顺序注册中间件：

```typescript
// app.ts
import express from 'express';
import 'express-async-errors';
import { responseFormatter } from './middlewares/responseFormatter';
import { errorHandler } from './middlewares/errorHandler';
import routes from './routes'; // 你的路由

const app = express();

// 基础中间件
app.use(express.json());
app.use(express.urlencoded({ extended: true }));

// 统一响应格式中间件（必须在路由之前）
app.use(responseFormatter);

// 路由
app.use('/api', routes);

// 404 处理（可选）
app.use((req, res) => {
  res.error(404, 'Not Found');
});

// 全局错误处理中间件（必须在路由之后）
app.use(errorHandler);

export default app;
```

---

## 6. 在路由中使用

现在在路由中可以直接使用 `res.success` 和 `res.error`，并且抛出错误会自动被全局错误处理中间件捕获。

```typescript
// routes/user.ts
import { Router } from 'express';
import { BusinessError } from '../middlewares/errorHandler';

const router = Router();

// 模拟获取用户信息
router.get('/user/:id', async (req, res) => {
  const userId = req.params.id;
  if (userId === '0') {
    // 抛出业务异常
    throw new BusinessError(1001, '用户不存在');
  }
  // 正常返回
  const user = { id: userId, name: 'John Doe' };
  res.success(user, '获取用户成功');
});

// 或者直接使用 next 传递错误
router.post('/user', async (req, res, next) => {
  try {
    // 某些异步操作可能抛出错误
    await someAsyncFunction();
    res.success(null, '创建成功');
  } catch (err) {
    next(err); // 传递给错误处理中间件
  }
});

export default router;
```

由于我们使用了 `express-async-errors`，在异步路由中直接 `throw` 也能被捕获，无需手动 `next`。

- 成功响应：

```json
{
  "code": 0,
```

```json
{
```

- 系统错误响应（开发环境带 error 详情）：

```json
{
  "code": 500,
  "message": "Internal Server Error",
  "error": "Cannot read property 'name' of undefined"
}
```

## 8. 总结

通过以上步骤，我们实现了：

- 统一的成功和错误响应格式。
- 扩展了 `Response` 对象，使用便捷。
- 全局错误处理中间件，集中处理业务异常和系统异常。
- 处理了未捕获异常和未处理的 Promise rejection，增强应用健壮性。
- 完全支持 TypeScript 类型提示。

这套方案可以轻松集成到任何 Express 项目中，让你的 API 输出规范统一，错误处理简洁高效。
